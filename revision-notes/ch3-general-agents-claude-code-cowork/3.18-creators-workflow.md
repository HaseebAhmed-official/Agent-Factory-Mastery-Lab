## 1. Key Vocabulary

| Term | Definition |
|------|-----------|
| **Creator's Workflow** | The set of production-level best practices developed by Boris Cherny (Claude Code's creator) and the Anthropic team for expert-level Claude Code usage |
| **Parallel Sessions** | Running multiple independent Claude Code conversations simultaneously, each with its own isolated context window, to work on separate tasks without cross-contamination |
| **Git Worktree** | A Git feature that lets you check out multiple branches into separate directories simultaneously, enabling parallel Claude sessions on different features of the same repo |
| **Plan Mode** | A Claude Code mode (activated via Shift+Tab twice) where Claude discusses and refines an approach *before* executing, reducing wasted iterations |
| **Claude-Reviews-Claude** | A two-session pattern where one Claude session creates work and a separate session (with fresh context) reviews it critically |
| **Self-Writing CLAUDE.md** | The practice of asking Claude to update its own CLAUDE.md after making a mistake, converting errors into permanent institutional memory |
| **Notes Directory Pattern** | Maintaining a `notes/` directory per project with per-feature documentation, referenced from CLAUDE.md as on-demand context rather than inline content |
| **@.claude Team Pattern** | Tagging code review comments with `@.claude` on GitHub to trigger CLAUDE.md updates, converting individual review feedback into organizational learning |
| **Session-End Review** | A skill invoked manually at session close to capture decisions, open questions, and insights while context is still fresh |
| **Correction Spiral** | A failure pattern where you keep correcting Claude on the same issue repeatedly instead of clearing context and starting fresh |
| **Context Pollution** | Irrelevant or failed information accumulating in the context window, degrading Claude's performance -- the root cause of most expert-identified failures |
| **Kitchen Sink Session** | A failure pattern where multiple unrelated tasks accumulate in one session, causing cross-contamination and context pollution |
| **PostToolUse Hook** | An event-driven automation that runs a command after Claude uses a specific tool -- e.g., auto-formatting after every file write |
| **Voice Dictation** | Using speech-to-text (fn twice on macOS) to compose prompts 3x faster than typing, enabling more detailed instructions |
| **Skill Portfolio** | A reusable collection of agent skills stored globally (`~/.claude/skills/`) or in a shared repo, applicable across projects |
| **Subagent** | An isolated Claude instance spawned by the main Claude session to handle a specific task in its own context window, reporting findings back without polluting the parent context |
| **code-simplifier** | A custom subagent that cleans up code after Claude completes an implementation |
| **verify-app** | A custom subagent providing detailed end-to-end testing instructions and verification |
| **build-validator** | A custom subagent that validates builds before merging to prevent broken merges |
| **code-architect** | A custom subagent that performs architecture review on complex changes |
| **"use subagents" directive** | Appending "use subagents" to any prompt, causing Claude to deploy multiple parallel subagents tackling different aspects simultaneously |
| **Challenge Prompts** | Asking Claude to critique/attack work rather than create it, activating a different analytical mode that surfaces flaws, inconsistencies, and gaps |
| **Elegant Solution Prompt** | "Knowing everything you know now, scrap this and create the elegant solution" -- leverages accumulated context from a first attempt for a superior second attempt |
| **Detailed Brief** | A specific, constraint-rich prompt with audience, success criteria, and boundaries that directly improves output quality by reducing ambiguity |
| **Conversational Data Interface** | Using Claude Code + MCP/CLI to query databases and data sources through natural language instead of learning query languages |

## 2. The Fundamental Constraint: Context Window

### What It Is

The context window fills rapidly and performance degrades as it fills. Every message, file read, and command output consumes tokens. This single constraint unifies ALL Claude Code best practices in this lesson.

### Why It Matters

This is the "why" behind every practice. Without understanding this constraint, the practices look like arbitrary rules. With it, they become logical consequences of a single physical limitation.

### How It Works

```
Context Window Lifecycle:

  FRESH          PRODUCTIVE        DEGRADING        POLLUTED
  ┌───┐          ┌───────┐         ┌─────────┐      ┌──────────┐
  │   │          │███    │         │████████ │      │██████████│
  │   │          │███    │         │████████ │      │██████████│
  │   │          │███    │         │████████ │      │██████████│
  └───┘          └───────┘         └─────────┘      └──────────┘
  0%              ~30%              ~70%              ~90%+
  "Ready"        "Sweet spot"      "/clear soon"    "Start over"
```

### Where It Fits

Connects to LLM Constraint 3: Limited Context Window (Ch 1, L1.2). The Creator's Workflow is the operational response to that constraint at expert scale.

### What Can Go Wrong

Ignoring context budget leads to every failure pattern in this lesson: Kitchen Sink Sessions, Correction Spirals, Infinite Exploration, and Trust-Then-Verify Gaps all trace back to context mismanagement.

## 4. Principle 1: Context is the Constraint

### What It Is

Everything flows from context window management. Why parallel sessions? Small focused contexts. Why `/clear`? Reset polluted context. Why subagents? Investigate in isolated contexts and report summaries.

### Key Practices

- **`/clear` between unrelated tasks**: Like clearing a kitchen counter between courses
- **`/statusline`**: Always-visible context consumption monitor
- **10-20% session abandonment**: Expert-level behavior, not failure. Starting fresh beats fighting polluted context

### When to `/clear`

After 2+ failed corrections on the same issue. Context is now cluttered with failed approaches that actively mislead Claude.

### Analogy

Counter space in a professional kitchen. Cluttered counter = mistakes. Clear station between courses. Don't pile all prep on one counter when you have three.

## 5. Principle 2: Parallelization Over Optimization

### What It Is

Boris Cherny calls parallel sessions "the single biggest productivity unlock." Multiple concurrent sessions, each with its own isolated context window.

### How It Works

```
ANTI-PATTERN: Sequential Single Session
┌──────────────────────────────────────────┐
│ Session 1                                 │
│ Task A → Task B → Task C → Task D        │
│ [context fills]  [degrades]  [polluted]   │
└──────────────────────────────────────────┘

BEST PRACTICE: Parallel Isolated Sessions
┌────────────┐  ┌────────────┐  ┌────────────┐
│ Session 1   │  │ Session 2   │  │ Session 3   │
│ Auth feature│  │ Bug fix     │  │ Analysis    │
│ [focused]   │  │ [focused]   │  │ [focused]   │
└────────────┘  └────────────┘  └────────────┘
```

### Implementation (Developers)

```bash
git worktree add ../auth-feature feature/auth
git worktree add ../bugfix-api bugfix/api-error
cd ../auth-feature && claude   # Terminal 1
cd ../bugfix-api && claude     # Terminal 2
```

### Implementation (Knowledge Workers)

Multiple browser tabs or Cowork workspaces. Organize by project/client/task.

### Pro Tips

- Shell aliases (`za`, `zb`, `zc`) for worktree switching
- Dedicated "analysis" worktree for logs and queries
- Color-code terminal tabs by task
- Start with 3 sessions before scaling

### Connection

O(1) scaling principle from L3.1 applied to context management. Each task gets O(1) context cost because it has a dedicated window.

### Friction Analysis

Without parallel sessions: working on a feature, teammate pings about a bug, you investigate in the same session. Context now has feature code AND bug investigation tangled together. Claude starts confusing the two. 30 minutes lost untangling.

## 6. Principle 3: Plan Mode Discipline

### What It Is

Always activate Plan Mode for non-trivial tasks. Shift+Tab twice. Iterate until alignment, then switch to auto-accept for execution.

### The Workflow

```
  State Goal → Plan Mode (Shift+Tab ×2) → Iterate Plan → Auto-accept → Execute
```

### Why This Works

Time invested in planning is 10x cheaper than time fixing misunderstood execution. Context consumed by planning (plain text discussion) is far less than context consumed by wrong execution (file reads, code generation, corrections).

```
WITHOUT PLAN MODE:
  You: "Add auth"
  Claude: [2000 tokens of session-based code]     ← WRONG APPROACH
  You: "No, use JWT"
  Claude: [1500 tokens undoing + 2000 JWT code]
  You: "That broke the middleware"
  Claude: [1000 tokens fixing]
  Total: 6500+ tokens, possibly still broken

WITH PLAN MODE:
  You: "Add auth" → Plan Mode
  Claude: "I'll use session-based auth because..."  ← 200 tokens
  You: "No, use JWT because we need stateless"
  Claude: "Updated plan: JWT with..."               ← 200 tokens
  You: "Looks good, execute"
  Claude: [2000 tokens of correct JWT code]
  Total: 2450 tokens, correct first time
```

### Recovery Pattern

If execution goes sideways, switch BACK to Plan Mode and re-plan. Don't push through confused execution.

### The Claude-Reviews-Claude Pattern

```
Session A (Writer)                Session B (Reviewer)
┌──────────────────┐             ┌──────────────────┐
│ Create plan in   │             │ "You are a staff │
│ Plan Mode        │──paste──→   │ engineer. Review  │
│                  │             │ for edge cases,   │
│                  │  ←paste──   │ security, gaps"   │
│ Incorporate      │             │                   │
│ feedback         │             │ Fresh context =   │
└──────────────────┘             │ catches blind     │
                                 │ spots"            │
                                 └──────────────────┘
```

Benefits: Fresh context catches blind spots. Different persona surfaces different concerns. Two-pass verification before code executes. Prevents sunk-cost bias.

## 7. Principle 4: Self-Evolving Documentation

### The Magic Phrase

> "Update your CLAUDE.md so you don't make that mistake again."

### How It Works

```
ERROR → CORRECTION → SELF-WRITE RULE → NEVER REPEAT

Example:
  Claude: import { auth } from "utils/auth"     ← WRONG
  You:    That should be '@/utils/auth'.
          Update your CLAUDE.md so you don't make that mistake again.
  Claude adds to CLAUDE.md:
          ## Import Paths
          - Always use @/ path alias
          - Never use relative 'utils/auth'
```

### Why Self-Writing Works

Claude understands the exact mistake context better than you could describe it. The rule it generates is precise and immediately testable.

### The Notes/ Directory Pattern (Deep Dive)

Instead of cramming everything into CLAUDE.md, maintain a separate `notes/` directory. CLAUDE.md points at notes rather than containing them.

```
ANTI-PATTERN: Everything in CLAUDE.md
┌─────────────────────────────┐
│ CLAUDE.md (800 lines)        │
│ ████████████████████████████ │  ← Loaded EVERY session
│ Most of it irrelevant        │  ← Signal buried in noise
└─────────────────────────────┘

BEST PRACTICE: CLAUDE.md + Notes Directory
┌──────────────┐   ┌──────────────────┐
│ CLAUDE.md     │   │ notes/           │
│ (50 lines)    │──→│ auth.md          │  ← Loaded ONLY when
│ Points at     │   │ billing.md       │     working on that
│ notes/        │   │ performance.md   │     specific area
└──────────────┘   └──────────────────┘
```

**The compounding mechanism**: After every PR, update notes. Over months, each notes file becomes comprehensive institutional knowledge that no single person remembers fully. 30-second additions compound into invaluable project-specific intelligence.

**Architecture for scaling**: When notes/ grows to 12+ files and 3,000+ lines, CLAUDE.md must include an **index table** telling Claude which file to read for which topic. This implements the Three-Zone Strategy (Ch 4):

```
Zone 1 (Always loaded):  CLAUDE.md with index → ~150 tokens
Zone 2 (On-demand):      Specific notes/ file  → ~500-800 tokens
Zone 3 (Never loaded):   Irrelevant notes/     → 0 tokens
```

### The @.claude Team Pattern (Deep Dive)

During GitHub code reviews, when a reviewer spots something Claude should have done better, they tag with `@.claude`. This triggers a CLAUDE.md update.

```
TRADITIONAL CODE REVIEW:
  Reviewer → Developer A learns → Developer B makes same mistake
  Knowledge: trapped in one person's memory

@.CLAUDE PATTERN:
  Reviewer → CLAUDE.md updated → ALL future Claude sessions fixed
  Knowledge: encoded in infrastructure, benefits entire team
```

**What Goes Wrong with @.claude**:

| Failure | Fix |
|---------|-----|
| Rule bloat | Only @.claude for patterns that repeat |
| Contradictory rules | Designate a CLAUDE.md owner |
| Stale rules | Review quarterly; delete rules Claude already follows |
| Missing context | Always include file paths and examples |

### Synthesis: Individual + Team Compounding

```
INDIVIDUAL (notes/ pattern):  Growth LINEAR with your usage
TEAM (@.claude pattern):      Growth MULTIPLIED by team size
COMBINED:                     Claude becomes most knowledgeable "team member"
```

## 8. Principle 5: Verification Infrastructure

### The Core Insight

Boris: "Probably the most important thing to get great results out of Claude Code: give Claude a way to verify its work. If Claude has that feedback loop, it will 2-3x the quality of the final result."

### How It Works

```
WITHOUT VERIFICATION:            WITH VERIFICATION:
  Claude writes → Done            Claude writes → Tests → Fails →
  (hope it works)                 Claude fixes → Tests → Pass → Done
                                  (OODA loop drives convergence)
```

### Forms of Verification

- Test suites (TDD)
- Browser automation (Claude Chrome extension tests UI)
- PostToolUse hooks (auto-format after write)
- Subagents reviewing output
- Bash commands checking results

### Connection

OODA Loop (L3.1): Verification enables the Observe step. Without it, the loop is broken. Also connects to Ch 6, Principle 3: Verification as Core Step.

## 9. Specialized Subagents for Common Workflows

### What It Is

Boris maintains a library of purpose-built subagents for workflows he does on most PRs:

| Subagent | Purpose |
|----------|---------|
| `code-simplifier` | Cleans up code after implementation |
| `verify-app` | Detailed end-to-end testing instructions |
| `build-validator` | Validates builds before merging |
| `code-architect` | Architecture review for complex changes |

### Why Subagents: Context Isolation

When Claude investigates your codebase, every file read consumes main context tokens. Subagents explore in isolated contexts and report summaries.

```
WITHOUT SUBAGENTS:
  Main context: 940 lines of file reads consumed → less budget for actual work

WITH SUBAGENTS:
  Subagent reads 940 lines → reports 10-line summary → main context gets insight
  without the cost. 99% context savings.
```

### The "use subagents" Directive

Appending "use subagents" to any prompt spawns multiple parallel subagents:

```
"Investigate auth AND check for OAuth utilities. Use subagents."
  → 2 subagents exploring simultaneously
  → Each reports findings
  → Context cost: LOW (only summaries)
  → Time cost: HALVED (parallel execution)
```

### Investigation Pattern

When Claude needs to research codebases, use subagents to keep main conversation clean. Subagents explore in isolated contexts, report findings.

### Permission Routing Through Hooks

Subagent permission requests can be routed through an Opus 4.5 hook that scans for suspicious operations. Safe operations auto-approve; dangerous ones flag for human review. Tiered autonomy system.

### Friction Analysis

Without subagents for investigation: 3 investigation rounds per session = 30% of context budget burned on exploration before doing any actual work. With subagents: same investigation, 99% context savings.

### Edge Cases

| Situation | Use Subagent? | Why |
|-----------|--------------|-----|
| Quick single-file lookup | **No** | Subagent overhead exceeds direct read cost |
| Deep multi-file investigation | **Yes** | Classic subagent use case |
| Writing code after investigation | **No** | Code writing needs main context for iteration |
| Parallel investigations of 3 systems | **Yes** | Three parallel explorers, three summary reports |
| Running tests | **No** | Test output needed in main context for Claude to fix failures |

### Adversarial Thinking

*"Subagents can hallucinate findings."* -- Valid. Mitigate with trust calibration:

```
Low stakes (code structure):    Trust subagent report directly
Medium stakes (implementation): Trust report, verify key references
High stakes (security, data):   Verify every claim against source files
```

### What Goes Wrong

| Failure | Fix |
|---------|-----|
| Subagent hallucination | Verify critical findings before acting |
| Over-delegation | Only use subagents when 3+ file reads needed |
| Unbounded exploration | Always scope: "Investigate X in files Y and Z" |
| Report too vague | Ask for specific outputs: file paths, line numbers, snippets |

## 10. Level Up Your Prompting

### Technique 1: Challenge Prompts

Ask Claude to ATTACK rather than create. Different persona = different concerns surfaced.

| Domain | Challenge Prompt |
|--------|-----------------|
| Developer | "Grill me on these changes and don't merge until I pass your test." |
| Writer | "Challenge every claim in this draft. What's unsupported?" |
| Strategist | "Poke holes in this plan. What am I missing?" |
| Analyst | "Prove to me this conclusion is correct. Show me the logic chain." |

**Why it works**: Generate mode optimizes for coherence. Challenge mode activates analytical mode looking for flaws. This is Claude-Reviews-Claude applied within a SINGLE session.

**Connection**: Operationalizes the "What Goes Wrong" framework (Misapplication, Omission, Excess, Interaction Failure).

### Technique 2: The Elegant Solution Escape

**The magic prompt**: "Knowing everything you know now, scrap this and create the elegant solution."

**When to use**: After Claude gives something that works but feels mediocre. The first attempt is NOT wasted -- it's context investment. Claude learned constraints, rejections, edge cases. The elegant solution prompt unlocks accumulated understanding into a fresh implementation.

**Critical**: Only works AFTER accumulated context. Using it on attempt 1 just gets a different mediocre answer.

```
Attempt 1: Limited understanding → correct but generic
  ▼ (through conversation, corrections, discussion)
Context accumulates: constraints, rejections, edge cases
  ▼ "Scrap this and create the elegant solution"
Attempt 2: Leverages ALL accumulated context → superior result
```

### Technique 3: Reduce Ambiguity Through Detailed Briefs

Specificity of input directly correlates with quality of output.

```
VAGUE: "Write a login page"
  → Generic login page

SPECIFIC: "Write a login page for our B2B SaaS.
  Must support SSO via SAML. Max 2 form fields.
  User-friendly errors. WCAG 2.1 AA accessible.
  Match design system in /styles."
  → Precisely tailored implementation
```

**Voice dictation connection**: Speaking is 3x faster than typing. Removes the typing bottleneck that causes under-specified prompts.

### Friction Analysis

Vague prompt = 1 min writing + 4 correction rounds (20 min) = 21 min total. Specific prompt = 5 min writing + 0 corrections = 5 min. Voice dictation = 1.5 min speaking + 0 corrections = 1.5 min.

### Edge Cases

| Technique | When It DOESN'T Work |
|-----------|---------------------|
| Challenge prompts | Early brainstorming where criticism kills ideas |
| Elegant solution | First attempt (no accumulated context yet) |
| Detailed brief | Exploratory tasks where constraints are unknown |
| Voice dictation | Noisy environments, code-heavy prompts needing exact syntax |

### What Goes Wrong

| Failure | Fix |
|---------|-----|
| Challenge without creation | Challenge AFTER creation, not instead |
| Premature elegant solution | Wait until context has accumulated |
| Over-specified brief | Distinguish hard constraints (must) from preferences (should) |
| Challenge fatigue | Scope: "Top 3 most critical issues only" |

## 11. Research and Data Analysis

### What It Is

Claude Code transforms into a conversational data interface. Describe what you want to know; Claude figures out how to get it. Replaces the need for query languages.

Boris: "Personally, I haven't written a line of SQL in 6+ months."

### The Paradigm Shift

```
TRADITIONAL: Question → Learn query language → Write query → Debug → Format → Interpret
CLAUDE CODE:  Question → Claude writes query → Formats → Interprets → Answer
```

### Domain-Specific Applications

| Role | What Claude Does | Example |
|------|-----------------|---------|
| Developer | Query databases via CLI | "How many API errors in the last 24 hours by endpoint?" |
| Analyst | Pull metrics, generate reports, visualizations | "Revenue trends by region for Q4" |
| Researcher | Search documents, synthesize, find contradictions | "Find contradictions across these 5 papers" |
| Knowledge Worker | Query connected tools via MCP | "Summarize last week's Slack decisions" |

### The Underlying Pattern

If data access exists through CLI, MCP, API, or files, Claude can query it conversationally.

### Skill-Building Pattern

Create a skill encoding HOW to access your data sources. Then anyone queries conversationally regardless of technical skill.

```yaml
# .claude/skills/query-metrics/SKILL.md
---
name: query-metrics
description: Query our analytics database
---
Access: psql -h analytics.internal -U readonly -d metrics
Key tables: events, users, revenue, subscriptions
Rules: LIMIT 1000, readonly user, markdown table output, include SQL query
```

### Product Overhang

Data access is a Product Overhang (L3.1): Existing Capability (database) + Missing Interface (natural language query) = Massive Unlock (anyone can access data). Claude Code IS the missing interface.

```
BEFORE: Data → [SQL wall] → Only 10% of staff can query
AFTER:  Data → [Claude + skill] → Anyone queries conversationally
```

### Friction Analysis

Traditional: PM asks analyst, joins queue, 2 days later gets answer, has follow-up, 3 more days. Total: 5 days. With Claude + data skill: 2 minutes for both questions.

### Edge Cases

| Situation | Works? | Why |
|-----------|--------|-----|
| Simple SELECT queries | Yes | Claude writes clean SQL |
| Complex multi-join analytics | Usually | Verify output |
| Real-time streaming data | No | Point-in-time queries only |
| Sensitive PII data | Carefully | Data passes through Claude's context |
| Unstructured data (audio/video) | Limited | Text extraction only |

### Adversarial Thinking

*"Non-technical users could run expensive scans or act on wrong numbers."*

Safety layers: (1) READONLY database user, (2) LIMIT defaults, (3) Separate analytics replica, (4) Always show raw query, (5) Schema in skill instructions, (6) Verify critical decisions with analyst.

### What Goes Wrong

| Failure | Fix |
|---------|-----|
| Hallucinated queries | Include schema in skill |
| Destructive queries | Enforce readonly database user |
| Over-trusting results | Always include raw query for verification |
| Performance disasters | LIMIT defaults, "never query production" |
| Sensitive data exposure | Database-level column restrictions |

## 13. Autonomous Problem Solving

### Core Pattern

Give Claude the problem, not the solution. Freedom to investigate often yields better approaches than prescribed steps.

```
PRESCRIPTIVE:                        OUTCOME-BASED:
"Add try-catch to lines 45-52"       "Login crashes silently on bad
                                      credentials. Fix it."
Claude does exactly what you said.   Claude might find the actual root
Might miss root cause.               cause, add validation, tests, UX.
```

### Domain Applications

| Domain | Approach |
|--------|----------|
| Developer | Paste Slack bug thread: "fix." Or: "Go fix failing CI tests." |
| Knowledge Worker | Paste confusing email: "draft a response that resolves this." |
| Researcher | Share 5 sources: "Find contradictions and synthesize truth." |
| Manager | Share stakeholder feedback: "Create a plan addressing everyone's concerns." |

### Principle

Describe desired outcomes rather than prescribing steps. The Orchestrator judges outcomes; the agent chooses methods.

### MCP Integration

Enable MCP integrations (Slack, Google Drive, Notion) so Claude pulls context directly -- zero context switching.

## 14. Model Selection

### Expert Recommendation

Boris: "I use Opus 4.5 with thinking for everything."

### Why

Wrong fast answers cost more time than right slow answers. Opus requires less correction/iteration, making total task completion faster despite slower per-response times.

```
SONNET: Response 1 (fast, wrong) → Correction → Response 2 (still off) →
        Correction → Response 3 (mostly right) → Fix → Done
        Wall clock: 8 min | 6 exchanges

OPUS:   Response 1 (slow, right) → Minor tweak → Done
        Wall clock: 4 min | 2 exchanges
```

The metric is total task completion time, not per-response speed. Fewer iterations = less context pollution.

## 15. Session Management Toolkit

```
CONTROL FLOW:
  Working normally
  ├── Stop mid-action? → Esc (preserves context)
  ├── Undo last exchange? → Esc+Esc or /rewind (restores code AND conversation)
  ├── Done with task? → /clear (reset context)
  ├── Switching topics? → /clear (don't mix workstreams)
  ├── 2+ failed corrections? → /clear + rewrite prompt with learnings
  ├── Closing for day? → /session-review skill first, then close
  └── Resuming tomorrow? → claude --continue (most recent) or --resume (pick)
```

### Session Naming

`/rename` for descriptive names ("oauth-migration", "billing-v2"). Sessions are like branches.

### When to Abandon

10-20% session abandonment is normal. When context is too polluted to recover, starting fresh with better prompts beats fighting through confusion.

## 20. Anti-Patterns and Failure Modes

### New Anti-Patterns Introduced in This Lesson

| Anti-Pattern | Symptom | Root Cause | Fix |
|-------------|---------|-----------|-----|
| **Kitchen Sink Session** | Started one task, asked unrelated questions, context cluttered | Context pollution from mixed tasks | `/clear` between unrelated tasks |
| **Correction Spiral** | Corrected Claude 2x, still wrong, correcting again | Failed approaches poisoning context | After 2 failures: `/clear` + rewrite prompt |
| **Over-Specified CLAUDE.md** | Claude ignores instructions; rules get buried | Signal-to-noise violation | Ruthlessly prune -- delete rules Claude already follows |
| **Trust-Then-Verify Gap** | Plausible output that fails on edge cases | No verification infrastructure | Always provide verification methods |
| **Infinite Exploration** | Asked "investigate" without scoping; context fills with reads | Unbounded investigation | Scope narrowly OR use subagents |

### Meta-Pattern

Most failures stem from context pollution -- either excessive irrelevant information or failed approaches cluttering the conversation. When uncertain, start fresh.

### "What Goes Wrong" Analysis

| Lens | Application to Creator's Workflow |
|------|----------------------------------|
| **Misapplication** | Using Plan Mode for trivial tasks (overhead exceeds execution). Using subagents for single-file lookups. Using "elegant solution" prompt on first attempt. |
| **Omission** | Skipping Plan Mode for non-trivial tasks. Never writing CLAUDE.md rules. No verification loops. Not using parallel sessions. |
| **Excess** | 800-line CLAUDE.md. 10 parallel sessions at once. Subagents for everything. Over-specified briefs. |
| **Interaction Failure** | Plan Mode + Correction Spiral (plan was fine, execution context got polluted). Subagent report + hallucinated findings (main context acts on wrong data). Notes/ pattern + no index (Claude reads all 12 files). |

## 22. Quiz Bank: Self-Test Questions

### Question 1 (Explain-Back)
**Q**: Boris Cherny says context window is THE fundamental constraint. Pick any TWO practices from this lesson and explain HOW each specifically addresses the context window constraint. What would happen if you ignored the practice?

**Model Answer**: See Principle 1 and Principle 2 sections. Parallel sessions give each task its own full context budget (without: Kitchen Sink Session pattern). Plan Mode keeps discussion in cheap text phase before expensive execution (without: Correction Spiral from wasted code generation + corrections).

### Question 2 (Architecture)
**Q**: Your notes/ directory has 12 files totaling 3,000 lines after 6 months. A new developer joins. What problem might they hit, and how would you architect CLAUDE.md to handle it?

**Model Answer**: If CLAUDE.md says "read notes/", Claude reads ALL 12 files = 6,000-9,000 tokens consumed at session start (Context Stuffing). If CLAUDE.md says nothing, Claude never discovers notes (Context Starvation). Solution: CLAUDE.md contains an index table mapping topics to specific notes files. Claude reads only the relevant file. Implements Three-Zone Strategy: Zone 1 (CLAUDE.md index, ~150 tokens), Zone 2 (specific file, on-demand), Zone 3 (irrelevant files, 0 tokens).

### Question 3 (Subagents)
**Q**: Why does Boris use subagents for code-simplifier, verify-app, build-validator, and code-architect instead of asking his main session? Connect to the fundamental constraint.

**Model Answer**: Each review involves reading multiple files and generating analysis. In the main session, all file reads and analysis consume main context tokens. By the time 4 reviews are done, context is heavily polluted. As subagents, each gets its own fresh context window, explores freely, and reports a concise summary. Main context receives only the summaries (~200 lines total vs potentially 4,000+ lines of raw exploration).

### Question 4 (Prompting)
**Q**: The "elegant solution" prompt only works AFTER accumulated context. Why would using it on attempt 1 fail? What role does the "failed" first attempt play?

**Model Answer**: On attempt 1, Claude has minimal understanding of your constraints, preferences, and edge cases. "Scrap and redo" with no accumulated context just produces a different generic solution. The first attempt is context investment: through conversation, corrections, and discussion, Claude learns what you rejected and why, discovers edge cases, understands constraints. The "elegant solution" prompt unlocks this accumulated understanding into a fresh implementation unburdened by iterative patches.
