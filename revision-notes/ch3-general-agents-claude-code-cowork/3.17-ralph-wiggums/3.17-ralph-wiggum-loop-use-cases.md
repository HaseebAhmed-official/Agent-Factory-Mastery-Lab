## 1. The Selection Principle

> **Success must be measurable by automated tools, not human judgment.**

| Word | What It Really Means |
|------|---------------------|
| **Measurable** | A number, status code, or exact string -- not "looks good" or "feels right" |
| **Automated tools** | `tsc`, `eslint`, `pytest`, `curl` -- programs that return deterministic output |
| **Not human judgment** | If two reasonable humans could disagree on whether it's "done," Ralph Loop can't decide either |

### The Objectivity Spectrum

```
OBJECTIVE ◄──────────────────────────────────► SUBJECTIVE
(Ralph Loop works)                              (Ralph Loop fails)

"0 errors"    "Tests pass"    "Looks clean"    "Feels right"    "On brand"
     ✓              ✓              ✗                ✗               ✗
```

## 2. Good Use Cases

### 2.1 Framework Upgrades

**Example**: "Upgrade Next.js from 14 to 15 and fix all breaking changes"
**Promise**: `"npm run build successful"`

**Why it works**:
- Build command has exactly two outcomes: success or failure with specific error messages
- Each error points to a specific file and line -- Claude can locate and fix it
- Progress is monotonic (generally) -- error count goes down over iterations
- Migration guides define what the new equivalent patterns are

**Hidden assumption**: The project HAS a build step. No build tool = no deterministic verification.

**Edge case**: Some framework upgrades involve behavioral changes that don't cause build errors. The build passes but the app is broken at runtime. Ralph Loop can't catch this -- you need runtime/integration tests.

### 2.2 Test-Driven Refactoring

**Example**: "Refactor auth module to use JWT while keeping all 47 tests passing"
**Promise**: `"All 47 tests passing"`

**Why this is the BEST use case**:
- Tests are the perfect completion promise -- designed to verify correctness
- Refactoring scope is bounded by what the tests cover
- Claude makes incremental changes, runs tests, sees failures, fixes, repeats
- Test suite acts as a specification -- Claude knows what "correct" means

**Hidden trap**: Quality of loop output is bounded by quality of tests.
```
Comprehensive tests → High-quality autonomous output
Shallow tests       → Technically "passing" but possibly broken
No tests            → Ralph Loop is useless (no feedback signal)
```

### 2.3 Linting/Type Error Resolution

**Example**: Fix all TypeScript errors. Promise: `"tsc reports 0 errors"`

**Why it works**: Compiler errors are the most deterministic feedback possible. Each error has a specific code, file, line, and description. Fixes are usually mechanical.

**Highest gaming risk**: This is where Claude is most likely to "cheat" via `@ts-ignore`, `any` type, deleting code, or changing `strict: true` to `strict: false`.

### 2.4 Deployment Debugging

**Example**: "Deploy to staging until health check passes"
**Promise**: `"Health check: 200 OK"`

**Riskiest good use case because**:
- Involves external systems (servers, networks, databases)
- Failures might not be in YOUR code
- Each iteration involves actual deployments -- side effects in shared environments
- Only appropriate when errors are in YOUR code/config and staging is isolated

**NEVER use for production deployments.**

## 3. Poor Use Cases

### 3.1 Tasks Requiring Human Judgment

Strategy, aesthetics, business priorities, ethical considerations, brand alignment.

No string means "the UI looks professional." These are multi-dimensional assessments requiring human perception.

**Tricky middle ground**: "Make the code more readable." Partially objective (cyclomatic complexity, function length) but ultimately involves human comprehension. Ralph Loop would optimize for a proxy metric, not actual readability.

### 3.2 Exploratory Research

Open-ended discovery has no natural "done" state. Always another paper, perspective, source. If you set `"RESEARCH_COMPLETE"`, Claude outputs it after first pass even though deeper research would yield better results.

**The deeper issue**: Ralph Loop assumes a convergent task (error count → zero). Research is divergent (scope expands as you discover more). Fundamentally incompatible.

### 3.3 Creative Work

No ceiling, no objective measurement. Each iteration might be "better" by one axis but "worse" by another. No convergence point. Claude optimizing for a vague creative goal across many iterations produces bland, over-polished output -- like photocopying a photocopy.

### 3.4 Multi-Goal Tasks

**The single-promise bottleneck**:
```
Task: "Fix bugs, add dark mode, update docs"

Which completion promise?
  "BUGS_FIXED"?     → Stops before dark mode/docs
  "DARK_MODE_ADDED"? → Might stop with bugs unfixed
  "ALL_DONE"?       → Claude decides when "all" means "all" (subjective!)
```

The completion promise is a single channel. Multi-goal tasks need multi-channel verification. **Fix**: Break into separate loops, each with its own promise.

### 3.5 Tasks Requiring External Input

Claude needs an API key, a human decision, or a third-party approval. Ralph Loop has no "pause and ask" mechanism. The loop becomes an expensive way to generate failure messages.

```
Iteration 1:  Claude starts task
Iteration 2:  Needs API key from you
Iteration 3-20: Tries workarounds, all fail
Max iterations hit. $30 spent. Zero progress.
```

## 4. The Decision Tree

```
1. Can a CLI tool verify success? (tsc, pytest, eslint)
   NO → Don't use Ralph Loop

   YES ↓

2. Is there a SINGLE unambiguous success signal?
   NO → Break into sub-tasks first

   YES ↓

3. Expected iterations > 10?
   NO → Manual iteration is faster (setup overhead)

   YES ↓

4. Is the task convergent? (error count decreases)
   NO → Divergent tasks loop forever

   YES ↓

5. Can Claude complete it without asking you anything?
   NO → Pre-answer all decisions in prompt, or split

   YES ↓

6. Are side effects contained? (no shared systems)
   NO → Use with extreme caution + low max-iterations

   YES ↓

   ✓ USE RALPH LOOP
```

## 5. Convergent vs Divergent Tasks

| Property | Convergent (good for loops) | Divergent (bad for loops) |
|----------|---------------------------|--------------------------|
| Direction | Error count → zero | Scope expands |
| Example | 25 type errors → 18 → 12 → 0 | 1 question → 3 sub-questions → 15 findings |
| Termination | Natural end state exists | No natural end state |
| Ralph Loop? | Yes | No |

## 6. The Gaming Risk

Claude may "fix" problems by taking shortcuts that technically satisfy the completion criteria but don't actually solve the problem.

### Common Gaming Strategies

| Shortcut | What Claude Does | Why It's Bad |
|----------|-----------------|-------------|
| `@ts-ignore` | Suppresses errors instead of fixing types | Errors still exist, just hidden |
| `any` type | Bypasses type checking entirely | Defeats the purpose of TypeScript |
| `eslint-disable` | Suppresses warnings per-line | 83 warnings "fixed" by 83 disable comments |
| Deleting code | Removes the function that has the error | No error if no code |
| Changing config | Sets `strict: false` in tsconfig | All errors gone, all safety gone |

### Anti-Gaming Prompt Template

```
Fix all TypeScript errors. Rules:
- Do NOT add @ts-ignore or @ts-expect-error
- Do NOT use the 'any' type
- Do NOT modify tsconfig.json
- Do NOT delete functions or files
- Do NOT add eslint-disable comments
- Fix the actual issues with proper solutions
```

## 7. Multi-Loop Campaigns

For tasks too big or too architectural for a single loop, decompose into a **campaign** of focused loops.

### Architecture: Bottom-Up Ordering

Fix dependencies first, then dependent code, then integration.

### Example: Pydantic v1 → v2 Migration (Python/FastAPI)

```
Loop 1: Fix Pydantic Model Definitions (15 files)
  Promise: "MODELS_V2_CLEAN"    Max: 20
  → git commit checkpoint

Loop 2: Fix FastAPI Endpoint Files (10 files)
  Promise: "ENDPOINTS_V2_CLEAN"  Max: 15
  → git commit checkpoint

Loop 3: Fix Service / Orchestration Files (8 files)
  Promise: "SERVICES_V2_CLEAN"   Max: 15
  → git commit checkpoint

Loop 4: Full Integration Test Pass
  Promise: "FULL_SUITE_V2_PASS"  Max: 15
  → git commit checkpoint
```

### Campaign Principles

1. **Bottom-up order** -- fix foundation layers first
2. **Scope isolation per loop** -- each loop only touches its own layer
3. **Git checkpoints between loops** -- if Loop 3 fails, you only lose Loop 3
4. **Specific test paths per loop** -- `tests/test_models/` not the full suite (faster, isolated)
5. **Small blast radius** -- damage from any single loop is contained

### The Orchestrator Pattern

You do the CREATIVE work (designing architecture, making judgment calls). Ralph Loop does the MECHANICAL work (replicating patterns, fixing deterministic errors). This IS the Digital FTE philosophy: humans for judgment, agents for execution.
