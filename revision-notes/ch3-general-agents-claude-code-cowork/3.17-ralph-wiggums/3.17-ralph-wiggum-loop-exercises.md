## 1. Comprehension Questions & Model Answers

### Q1: Why was Ralph Loop built as a plugin, not built-in?

**Model Answer**: Deliberate opt-in vs accidental activation. If autonomous iteration were built-in, users could accidentally trigger unbounded loops -- racking up $100+ costs and making widespread codebase changes without realizing. As a plugin, it enforces three safety layers: (1) explicit installation, (2) conscious invocation with parameters, (3) progressive trust (beginners never encounter it). This matches Claude Code's philosophy of escalating autonomy with escalating deliberateness.

### Q2: False Positive Scenario

**Scenario**: Ralph Loop fixing all failing tests. Promise: `"ALL_TESTS_PASSING"`. After 15 iterations, loop exits with promise triggered. But 3 tests still fail manually. What went wrong?

**Model Answer -- Four Root Causes**:

| # | Root Cause | Explanation | Prevention |
|---|-----------|-------------|-----------|
| 1 | Claude hallucinated success | Believed tests passed without verification, output promise prematurely | Tie promise to specific tool output: "only output X if pytest shows '0 failed'" |
| 2 | Partial test execution | Ran only a subset of tests (e.g., one directory). Subset passed. | Specify exact command: `pytest` from root, no path filters |
| 3 | Environment drift | Claude modified environment (installed package, changed config) making tests pass in-session but not clean | Review ALL changes via `git diff` after loop exits |
| 4 | Flaky tests | Tests intermittently pass/fail due to timing/network/state. Passed during Claude's run, failed during yours. | Fix flaky tests before using autonomous iteration |

**Key takeaway**: The completion promise is trust-but-verify. ALWAYS review results after the loop exits.

## 2. Exercise 1: The Judgment Call

**Task**: Classify each scenario as Ralph Loop / Manual / Hybrid.

| # | Scenario | Verdict | Reasoning |
|---|----------|---------|-----------|
| A | Fix 8 failing unit tests after dependency update | **Manual** | Under 10 iterations. Setup overhead ≥ manual time. You want to see each failure to understand the new dependency. Exception: if all 8 are the same type of error, loop could work. |
| B | "Make the dashboard load faster" | **Manual** | Fails three criteria: no objective signal, divergent task, requires judgment (code splitting strategy, caching policy, etc.) |
| C | Update 200 import paths after moving `src/utils/` to `src/lib/` | **Ralph Loop** | Objective verification (`tsc --noEmit`), mechanical fixes, high iteration count, independent errors, no judgment needed. Near-perfect use case. |
| D | Write integration tests for 15 API endpoints | **Manual** | Generative task, not convergent. No starting error count. Tests Claude writes will "pass" trivially. "Tests pass" ≠ "tests are good." |
| E | Fix 35 ESLint errors (30 auto-fixable + 5 "no-any" violations) | **Hybrid** | 30 mechanical errors → Ralph Loop. 5 type-design errors → Manual (require understanding data model). Separate them. |
| F | Deploy to production and fix issues | **Manual** | NEVER use Ralph Loop for production. Shared system, irreversible side effects, infrastructure errors outside Claude's scope, maximum blast radius. |

## 3. Exercise 2: Prompt Surgery

**Task**: Find all problems in this badly written command and rewrite it.

### The Bad Command

```bash
/ralph-loop "Fix all the issues in the codebase and make it work properly.
Output DONE when finished." \
--completion-promise "done"
```

### Seven Problems Found

| # | Problem | Danger |
|---|---------|--------|
| 1 | No `--max-iterations` | No safety limit. Infinite loop possible. $200+ bill. |
| 2 | Case mismatch: prompt says `"DONE"`, flag says `"done"` | Exact string matching. `DONE` ≠ `done`. Loop never exits. |
| 3 | Vague task: "Fix all the issues" | Unbounded scope. Claude "fixes" things you never intended. |
| 4 | Vague success: "make it work properly" | "Properly" is subjective. No tool can verify it. |
| 5 | No verification command | Claude might think it's fixed without running tests/builds. |
| 6 | No scope constraints | Claude could modify configs, delete tests, install packages. |
| 7 | `DONE` is a common word | Could appear in Claude's progress narration, causing false exit. |

### The Corrected Version

```bash
/ralph-loop "Fix all TypeScript compilation errors in the src/ directory:
- Run: tsc --noEmit --pretty
- Read each error and fix the underlying type issue
- Do NOT use @ts-ignore, @ts-expect-error, or the 'any' type
- Do NOT modify tsconfig.json
- Do NOT modify files outside of src/
- Do NOT delete or rename existing functions
- After each round of fixes, re-run tsc --noEmit --pretty
- Output <promise>TSC_ZERO_ERRORS_ACHIEVED</promise> only when tsc output
  shows 'Found 0 errors'" \
--max-iterations 35 \
--completion-promise "TSC_ZERO_ERRORS_ACHIEVED"
```

### What Changed

| Fix | What Was Added |
|-----|---------------|
| Max iterations | `--max-iterations 35` |
| Case match | Promise identical in prompt and flag |
| Specific task | "TypeScript compilation errors" not "all issues" |
| Verification command | `tsc --noEmit --pretty` explicit |
| Scope boundaries | "only in src/", "don't modify tsconfig" |
| Anti-gaming rules | No @ts-ignore, no any, no deleting |
| Unique promise | `TSC_ZERO_ERRORS_ACHIEVED` won't appear accidentally |
| Promise tied to tool | "only when tsc shows 'Found 0 errors'" |

## 4. Exercise 3: Multi-Loop Campaign Design

**Scenario**: Pydantic v1 → v2 migration. 15 models, 10 FastAPI endpoints, 8 services, 60 tests.

### Campaign Architecture

```
Loop 1: Models → checkpoint → Loop 2: Endpoints → checkpoint
                                                      │
Loop 3: Services → checkpoint → Loop 4: Integration
```

### Loop 1: Fix Pydantic Model Definitions

```bash
/ralph-loop "Migrate all Pydantic models from v1 to v2 syntax:
- Run: python -m pytest tests/test_models/ -x --tb=short
- Change 'class Config:' to 'model_config = ConfigDict(...)'
- Change 'validator' to 'field_validator'
- Change 'root_validator' to 'model_validator'
- Change '.dict()' to '.model_dump()'
- Change '.json()' to '.model_dump_json()'
- Do NOT modify files outside models/
- Do NOT change business logic
- Output <promise>MODELS_V2_CLEAN</promise> when all model tests pass" \
--max-iterations 20 --completion-promise "MODELS_V2_CLEAN"
```

### Loop 2: Fix FastAPI Endpoints

```bash
/ralph-loop "Update all FastAPI endpoints for Pydantic v2:
- Run: python -m pytest tests/test_endpoints/ -x --tb=short
- Update response_model usage, .dict()/.json() calls
- Do NOT modify models/ or service files
- Output <promise>ENDPOINTS_V2_CLEAN</promise> when all endpoint tests pass" \
--max-iterations 15 --completion-promise "ENDPOINTS_V2_CLEAN"
```

### Loop 3: Fix Services

```bash
/ralph-loop "Update all service files for Pydantic v2:
- Run: python -m pytest tests/test_services/ -x --tb=short
- Fix .dict(), .json(), .parse_obj(), .from_orm() calls
- Do NOT modify models/ or endpoints/
- Output <promise>SERVICES_V2_CLEAN</promise> when all service tests pass" \
--max-iterations 15 --completion-promise "SERVICES_V2_CLEAN"
```

### Loop 4: Full Integration

```bash
/ralph-loop "Run complete test suite, fix remaining Pydantic v2 issues:
- Run: python -m pytest --tb=short
- Do NOT add pytest.skip or xfail markers
- Output <promise>FULL_SUITE_V2_PASS</promise> when pytest shows '60 passed'" \
--max-iterations 15 --completion-promise "FULL_SUITE_V2_PASS"
```

### Why This Ordering

- **Bottom-up**: Models are the foundation; endpoints and services depend on them
- **Scope isolation**: Each loop only touches its own layer
- **Git checkpoints**: If Loop 3 fails, only Loop 3's work is lost
- **Specific test paths**: Faster feedback, isolated verification

## 5. Exercise 4: Digital FTE Builder Scenarios

**Context**: Building 24/7 custom AI agents (Digital FTEs) with Python, FastAPI, Docker.

| # | Scenario | Verdict | Reasoning |
|---|----------|---------|-----------|
| A | Dockerfile failing with 12 pip dependency conflicts (Python 3.11→3.12) | **Ralph Loop** | `docker build .` gives deterministic output. Each conflict is a version pin fix. 12 conflicts likely cascade to 15-20 iterations. Hidden risk: some packages may not support 3.12 at all → same error repeating → cancel. |
| B | Add rate limiting to all 10 FastAPI endpoints | **Manual** | DESIGN task, not fix task. Requires decisions: per-user vs per-IP? Redis vs in-memory? What rate per endpoint? No starting error count. Generative, not convergent. Alternative: design on one endpoint manually, then Ralph Loop to replicate the pattern across 9 more. |
| C | 15 failing tests after changing LLM prompt template format | **Hybrid** | Two categories: mechanical (test assertions need updating) and judgment (new prompt produces different output -- is it acceptable?). Manual to separate them, Ralph Loop for the mechanical fixes, manual for quality judgments. |
| D | Improve agent response quality for customer complaints | **Manual** | Subjective, divergent, no convergence. Core prompt engineering work -- the most human-judgment-intensive part of FTE building. Can't outsource taste to a loop. |
| E | Docker Compose with 8 configuration errors | **Ralph Loop (cautious)** | `docker compose up` gives deterministic errors. Each fix is specific. Caution: some errors involve external state (port conflicts, stale volumes) that Claude can't fix without destructive actions. |
| F | Add structured JSON error responses to all 10 endpoints | **Hybrid** | Design the error schema and exception handlers manually (judgment). Implement on one endpoint as template. Ralph Loop to replicate across 9 more (mechanical). Review for endpoint-specific edge cases (judgment). |

## 6. Bonus: Hidden Risk Analysis

**Scenario**: Express.js → Fastify migration, 120 endpoint tests.

**Appears to be**: Perfect Ralph Loop use case (test-driven refactoring, clear tests).

**Actually has four hidden risks**:

| Risk | What Goes Wrong |
|------|----------------|
| Tests verify Express-specific behavior | Claude makes Fastify mimic Express (wrong) or changes tests (scope creep) |
| Too architectural for line-by-line fixing | Claude creates Franken-hybrid of Express wrappers inside Fastify |
| Blast radius too large | Every API file touched. Error at iteration 10 propagates through 11-50 |
| Coupled errors | Fixing one middleware affects all routes using it. Cascading fix spiral likely. |

**Solution**: Decompose into a multi-loop campaign with small scope per loop and manual review checkpoints between each.
