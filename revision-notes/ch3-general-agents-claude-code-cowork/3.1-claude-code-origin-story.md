## 1. Key Vocabulary

| Term | Definition |
|------|-----------|
| **Claude Code** | Anthropic's terminal-based (CLI) AI coding agent. Runs in your command line. Can read files, write code, run commands, and interact with your entire project. |
| **Claude Cowork** | Anthropic's desktop-based AI agent interface. A graphical (visual) way to interact with Claude for knowledge work, documents, and browser tasks. |
| **CLI** | **Command-Line Interface** -- a text-based way to interact with your computer by typing commands (Terminal, Command Prompt, bash). |
| **General Agent** | An AI agent that handles a wide variety of tasks across domains without being custom-built for one specific workflow. Claude Code is the prime example. |
| **Custom Agent** | An AI agent optimized for a specific, repetitive workflow. Requires weeks to develop. Built BY General Agents. |
| **Agentic AI** | AI systems that don't just answer questions -- they take *actions*: reading files, running commands, making decisions, completing multi-step tasks autonomously. |
| **Product Overhang** | A situation where an AI model already possesses a capability, but lacks the product interface to express it. The capability is "locked inside." |
| **OODA Loop** | **Observe-Orient-Decide-Act** -- a decision-making cycle used by agentic AI to iteratively solve problems. Repeats until the goal is achieved. |
| **Filesystem Access** | The ability of an AI agent to directly read, write, and manipulate files on your computer -- the key unlock that created Claude Code. |
| **Permission Loop** | Claude Code's safety mechanism: it asks for your approval before every file write, delete, or command execution. |
| **Agent Skills** | Encoded expertise -- documented procedures that teach Claude precise methods, reasoning patterns, or domain knowledge. Intelligence plugins you create and can monetize. |
| **Digital FTE** | **Digital Full-Time Equivalent** -- an AI agent that delivers value equivalent to a human employee in a specific domain. |
| **Orchestrator** | The new role of the human developer: directing AI agents, providing judgment, reviewing output -- rather than typing code manually. |

## 3. Product Overhang

### Definition

A Product Overhang exists when an AI model already possesses a capability but has no product interface to express it.

### The Formula

```
Capability the model ALREADY has
+ Interface it DOESN'T have yet
──────────────────────────────────
= Massive unlock when someone builds the bridge
```

### Claude Code Example

- **Existing capability**: Claude could reason about code, understand project structures, trace logic
- **Missing interface**: Filesystem access (ability to read/write actual files)
- **Bridge built**: Boris Cherny's experiment giving Claude terminal access
- **Result**: A tool that writes 90% of its own codebase

### Why This Matters For You

As an orchestrator, your job includes **spotting Product Overhangs** -- recognizing where AI already has latent capability waiting for the right interface. This is a strategic skill, not just technical knowledge.

## 5. Passive vs Agentic Model

### Passive Model (Traditional AI Chat -- e.g., ChatGPT)

```
Describe problem → receive generic suggestion → copy-paste → adapt → test
```

- Human is the **execution layer**
- AI is just a **suggestion machine**
- Every step requires human action between AI responses
- AI never sees your actual code -- only fragments you choose to share

### Agentic Model (Claude Code)

```
Describe goal → AI reads actual files → proposes specific changes →
executes with approval → tests → iterates
```

- Human is the **judgment layer** (Orchestrator)
- AI is the **execution layer**
- OODA loop runs autonomously between human checkpoints
- AI sees your **real project** -- actual files, structure, conventions

### Side-by-Side Comparison

```

                    copy-paste

Who navigates       YOU                        THE AGENT
the codebase:

Who adapts the      YOU (different var names)   THE AGENT (reads your code)
fix to fit:

Who runs tests:     YOU (manually)             THE AGENT (automatically)

Who finds root      YOU (after multiple        THE AGENT (traces full
cause:              round trips)               call chain)

Your role:          Typist / messenger         Orchestrator / reviewer

Round trips:        6-8 per bug                1
```

### The Course Analogy

> **Pair programming vs. consulting over the phone.** One reviews your actual screen; the other doesn't.

### The Film Director Analogy

> A chatbot is like calling an actor and asking "how would you deliver this line?" Claude Code is like having that actor **on set** -- performing, adjusting, interacting with the full environment. The actor does the work; you provide direction and judgment.

## 8. The OODA Loop in Agentic Context

### The Four Steps

| Step | Meaning | Claude Code Example |
|------|---------|-------------------|
| **Observe** | Gather information | Reads files, runs tests, sees errors |
| **Orient** | Interpret and understand | Understands WHY something failed based on codebase context |
| **Decide** | Choose a course of action | Selects what to fix and which approach to use |
| **Act** | Execute the decision | Writes the fix, runs the test again |

### How It Runs in Claude Code

```
Human sets goal
     │
     ▼
OBSERVE: Agent reads existing code, runs tests, sees errors
     │
     ▼
ORIENT: Agent understands why something failed
     │
     ▼
DECIDE: Agent chooses what to fix and how
     │
     ▼
ACT: Agent writes fix, runs test
     │
     ▼
Tests pass? ──NO──→ Back to OBSERVE (loop repeats)
     │
    YES
     │
     ▼
Agent moves to next task
```

### Critical Insight

This loop repeats **hundreds of times** autonomously. A chatbot runs ONE cycle (you ask → it suggests) and then waits. Claude Code runs the full loop until the goal is achieved.

### The OODA Loop REQUIRES Feedback

Without tests, the Observe step has nothing to observe after Act. The loop becomes:

```
BROKEN LOOP: Write code → ??? → Assume it works → Move on → Bugs pile up
```

> **Every agentic workflow is a loop. Every loop needs feedback. No feedback = a blind agent.**

## 10. Anti-Patterns & Failure Modes

### Named Anti-Patterns Relevant to 3.1

| Anti-Pattern | What It Is | Why It's Dangerous |
|-------------|-----------|-------------------|
| **Unverified Trust** | Accepting agent output without reviewing it | Agent is probabilistic -- it can be wrong. Bugs slip through. |
| **Monolithic Decomposition** | Giving agent a massive task ("build the whole app") instead of small steps | Agent loses context, produces incoherent results. |
| **Staying Passive** | Using Claude Code like a chatbot -- just asking questions, not giving tasks | You miss the entire agentic advantage. You stay a typist. |
| **Over-Distrust** | Rewriting every agent output manually "to be safe" | Eliminates the agentic advantage. You're back to being the execution layer. |

### The Trust Spectrum

```
No trust              Balanced trust            Blind trust
(rewrite everything)  (review, approve)         (accept everything)
      │                     │                        │
      ▼                     ▼                        ▼
  Typist again         ORCHESTRATOR              Unverified Trust
  (no agentic value)   (maximum value)           (dangerous)
```

**The sweet spot: review output, don't rewrite it.**

## 12. OODA Loop as a Diagnostic Tool

When an agent produces bad results, diagnose WHERE the loop broke:

```
OODA DIAGNOSTIC:

  Agent doing wrong thing?         → Check ORIENT (misunderstanding)
  Agent not seeing something?      → Check OBSERVE (missing information)
  Agent seeing wrong info?         → Check OBSERVE (bad data source)
  Agent knows what to do but       → Check ACT (blocked by constraints)
    can't do it?
  Agent choosing wrong approach    → Check DECIDE (wrong strategy)
    despite understanding?
```

### Diagnostic Scenarios (Studied)

| Scenario | OODA Break Point | Why |
|----------|-----------------|-----|
| Claude writes camelCase but team uses snake_case | **Orient** (or Observe if conventions aren't documented) | Claude sees code but doesn't recognize the convention pattern |
| Error message points to wrong file | **Orient** | Claude observes the error but builds wrong mental model |
| Claude knows the fix but lacks write permission | **Act** | Observe, Orient, Decide all worked -- execution is blocked |
| Claude fixes a bug but introduces a new one elsewhere | **Observe** (then Orient) | Didn't observe related code; didn't understand blast radius |

### Key Insight

> The OODA Loop isn't just a concept to memorize -- it's a **diagnostic tool**. When something goes wrong with an agent, ask: "Where in the OODA loop did it break?" That's orchestrator thinking.

## 13. Scaling: Why Agentic Dominates at Complexity

### The Scaling Problem with Copy-Paste

```
Bug is 2 calls deep:    ~4 copy-paste round trips
Bug is 4 calls deep:    ~8 round trips
Bug is 8 calls deep:    ~16 round trips
Bug spans 3 files:      Multiply by 3

PLUS: ChatGPT's context window fills up after 8-10 code pastes.
It FORGETS the first files you shared. Now you're re-pasting
AND correcting its confusion.

Copy-paste time:        O(n) -- grows proportionally with depth
Copy-paste frustration: O(n²) -- grows FASTER (context loss)
```

### Agentic Scaling

```
Bug is 2 calls deep:    Agent traces 2 files. ~1 minute.
Bug is 4 calls deep:    Agent traces 4 files. ~2 minutes.
Bug is 8 calls deep:    Agent traces 8 files. ~3 minutes.

Agent time:    O(log n) -- barely grows
Your effort:   O(1) -- CONSTANT. You typed one prompt.
```

### The Graph

```
Effort │
  ▲    │        Copy-paste ──── /
       │                      /
       │                    /
       │                  /
       │                /
       │              /
       │     ────────────────── Agentic (nearly flat)
       │
       └──────────────────────────────► Complexity
```

### Strategic Takeaway

> Simple problems? Both approaches feel similar. **Complex problems?** The gap explodes. The more complex the task, the more the agentic approach dominates.

## 16. Adversarial Analysis: Is Filesystem Access Dangerous?

### Where The Safety Argument Is Valid

| Concern | Validity |
|---------|----------|
| Can read `.env` with API keys | **Real risk** -- secrets could leak into output |
| Can delete or overwrite files | **Real risk** -- bad decisions destroy work |
| Changes happen faster than you can track | **Partially valid** -- speed reduces visibility |

### Where The Argument Falls Apart

| Counter-Point | Why |
|--------------|-----|
| **Permission loops** | Claude Code asks before EVERY write/delete/command. Human approves each. |
| **Git safety net** | Every change is reversible. `git checkout`, `git revert`. Small Reversible Decomposition (Ch.6, Principle 4). |
| **Copy-paste has its own risks** | Pasting unverified code is also dangerous -- with NO automated testing step. |
| **Proves too much** | By this logic, junior devs shouldn't have filesystem access either. Solution = guardrails, not removal. |

### The Safety Spectrum

```
No access             Permission loops         Full autonomy
(chatbot)             (Claude Code)            (no approval)
    │                      │                        │
    ▼                      ▼                        ▼
Very safe              RIGHT BALANCE             Dangerous
Very slow              Productive                Fast but risky
No agentic value       Maximum value             Maximum risk
```
