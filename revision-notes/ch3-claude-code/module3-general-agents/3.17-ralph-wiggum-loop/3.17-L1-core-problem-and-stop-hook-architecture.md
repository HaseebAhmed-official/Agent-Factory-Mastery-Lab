## 2. The Iteration Fatigue Problem

### What It Is

Iteration fatigue is the hidden cost structure that accumulates when a task requires multiple rounds of human-mediated feedback between Claude and external tool output. The canonical example: resolving 47 linting errors.

### Why It Matters

Tasks requiring 10+ iterations represent the highest-value automation targets. Below that threshold, manual iteration is fast enough. Above it, three hidden costs compound:

| Cost Type | Description | Cognitive Load |
|-----------|-------------|----------------|
| **Waiting time** | Idle period while Claude processes each iteration | Breaks focus |
| **Context switching** | Attention bouncing between terminal output and chat interface | Interrupts flow |
| **Error transcription** | Copy-paste operations introduce noise, partial output, typos | Corrupts input |

### How It Works (Manual Loop — The Problem State)

```
Run tool (e.g., linter)
       ↓
Read output in terminal
       ↓
Copy error messages manually
       ↓
Paste to Claude
       ↓
Claude processes and fixes some errors
       ↓
Repeat ×10+ rounds
       ↓
[You are the loop]
```

The human operator is the connective tissue of the loop. Remove the operator, and the loop breaks.

### Where It Fits

This is the motivating problem for the entire Ralph Wiggum Loop plugin. The plugin's sole purpose is to replace the human-as-connector with a machine-as-connector.

### What Can Go Wrong

**Common Pitfalls**:
- ❌ **Applying Ralph Loop to <10 iteration tasks**: Overhead of setup exceeds manual effort
  - ✅ **Fix**: Reserve for tasks with high iteration counts or non-deterministic convergence time
- ❌ **Underestimating transcription errors**: Partial copies of output cause Claude to work from incomplete data
  - ✅ **Fix**: Use tool-output capture directly rather than clipboard operations

**Analogy**:
> A postal worker who delivers and retrieves every letter between two offices — the system only works if the worker shows up every time. Replace the worker with a pneumatic tube, and the loop runs itself.

## 3. Stop Hook Architecture

### What It Is

The Stop hook is the technical mechanism that enables autonomous looping. It is a Claude Code lifecycle hook that intercepts the `Stop` event — the moment Claude would ordinarily terminate the session.

### Why It Matters

Without a Stop hook, Claude's default behavior is: complete one pass, produce output, exit. Each task cycle is a discrete session with no memory of continuation intent. The Stop hook converts a one-shot session into a potentially infinite iterative loop with a conditional exit.

### How It Works

**Normal session flow (no hook)**:
```
Claude receives task
       ↓
Claude works on task
       ↓
Claude produces output
       ↓
Claude exits ← [session ends here]
```

**Session flow with Stop hook**:
```
Claude receives task
       ↓
Claude works on task
       ↓
Claude produces output
       ↓
Claude attempts to exit
       ↓
Stop hook fires ← [interception point]
       ↓
Hook reinjects prompt: "Does output contain completion promise?"
       ↓
    NO ──────→ Claude continues next iteration ──→ [loops back]
       ↓ YES
Claude exits ← [session ends here, task complete]
```

### Where It Fits

The Stop hook integrates with the broader Claude Code hook system covered in Lesson 3.15. It is a specialized application of the hook lifecycle to create autonomous control flow.

### What Can Go Wrong

**Common Pitfalls**:
- ❌ **Infinite loop (no `--max-iterations` set)**: Hook never allows exit if completion promise is never produced
  - ✅ **Fix**: Always set `--max-iterations`; start conservatively (5-10), increase if needed
- ❌ **Assuming the hook "watches" Claude during execution**: The hook only fires at the exit boundary, not mid-task
  - ✅ **Fix**: Design task prompts to produce output that can be evaluated at completion, not mid-process

**Visual Aid — Hook Interception**:
```
[Normal Exit]          [Hook Intercept]
     │                      │
     ▼                      ▼
 Claude done           Claude done
     │                      │
     ▼                 Stop hook fires
  Session                   │
   ends              ┌──────┴──────┐
                     │             │
               Promise        No promise
               found?          found?
                 │                 │
                 ▼                 ▼
            Allow exit       Reinject prompt
                             (continue loop)
```

**Analogy**:
> A factory quality control inspector stationed at the exit gate — not watching the assembly line, but checking every item before it leaves. Items that fail inspection go back into production; items that pass exit the facility.

## 4. Completion Promise Constraint

### What It Is

The completion promise is the loop's exit condition: a static string that must appear verbatim in Claude's output for the loop to terminate. Specified via `--completion-promise` parameter at launch.

### Why It Matters

The constraint governs all task design for Ralph Loop. Because the promise cannot change mid-run, and because only one promise can be active at a time, the success condition must be knowable upfront and expressible as a single string.

### How It Works

The system checks output using exact string matching:
- `"0 problems"` → passes only if output contains exactly that string
- `"All tests passing"` → passes only if output contains exactly that string
- Case-sensitive: `"DONE"` ≠ `"done"`

**Embedded Promise Pattern** (recommended):
```
/ralph-loop "Task description here
- Requirement 1
- Requirement 2
Output <promise>COMPLETION_MARKER</promise> when done."
--max-iterations 20 --completion-promise "COMPLETION_MARKER"
```

This pattern gives explicit control over the completion signal, avoiding reliance on variable tool output strings.

**Natural output pattern** (riskier):
```
--completion-promise "0 problems"
```
Relies on the linter output format never changing. Fragile if tool output varies.

### What Can Go Wrong

**Common Pitfalls**:
- ❌ **Using variable output strings as completion promise**: Tool output format may change across versions
  - ✅ **Fix**: Use Embedded Promise Pattern — Claude outputs a controlled marker, not a tool output string
- ❌ **Multiple success conditions**: Can't combine `"0 errors"` AND `"All tests passing"` simultaneously
  - ✅ **Fix**: Design a single composite marker: `"ALL_CHECKS_PASSED"`
- ❌ **Typo in promise string**: Loop never terminates even when task succeeds
  - ✅ **Fix**: Test completion promise with `--dry-run` before launching full loop

## 6. Anti-Patterns & Failure Modes

### Anti-Pattern 1: Unbounded Loop

**Pattern**: Launching Ralph Loop without `--max-iterations`, relying on completion promise to terminate.

**What Goes Wrong** (Four-Axis Analysis):
1. **Misapplication**: Treats the loop as self-terminating by design when it is only self-terminating if the completion promise is eventually produced
2. **Omission**: Leaves out the safety ceiling that prevents runaway execution
3. **Excess**: Allows unlimited Claude API calls, potentially accumulating $100+ in costs
4. **Interaction Failure**: If the task has a bug that prevents the completion promise from ever being produced, the loop runs until external intervention

**Fix**: Always include `--max-iterations`. Start at 10-15 for moderate tasks; increase based on empirical run data.

### Anti-Pattern 2: Fragile Natural Output Promise

**Pattern**: Using tool output strings (`"0 errors"`, `"All tests passing"`) as completion promises.

**What Goes Wrong**:
1. **Misapplication**: Assumes tool output format is stable across versions and environments
2. **Omission**: Doesn't account for output formatting differences (trailing newlines, whitespace, color codes)
3. **Excess**: None
4. **Interaction Failure**: Works in dev, fails in CI due to different tool version or output flag

**Fix**: Embedded Promise Pattern — Claude outputs a controlled, immutable string as the last action when task is complete.

## 7. Frameworks & Mental Models

### Framework 1: Iteration Fatigue Threshold

**Formula**:
```
IF expected_iterations >= 10 → Use Ralph Loop
IF expected_iterations < 10  → Manual iteration (lower setup overhead)
IF convergence_time = unknown → Use Ralph Loop (non-deterministic tasks)
```

**Application**: Apply before any multi-pass AI-assisted task to determine whether automation investment pays off.

---

### Framework 2: Stop Hook Interception Model

**Formula**:
```
[Task Start]
     ↓
[Claude Works]
     ↓
[Exit Attempt] → [Stop Hook] → [Completion Check]
                                    ↓          ↓
                               [Promise    [No Promise
                                Found]      Found]
                                   ↓            ↓
                              [Terminate]  [Reinject →
                                            Continue]
```

**Components**:
1. **Exit Attempt**: Claude's natural termination signal
2. **Stop Hook**: Intercepts exit, injects evaluation prompt
3. **Completion Check**: Exact string matching against `--completion-promise`
4. **Reinject**: Continues loop with next iteration prompt

---

### Framework 3: Promise Design Hierarchy

**Formula**:
```
Embedded Promise (controlled marker)     ← Most reliable
     > Tool output with stable format    ← Acceptable if format is guaranteed
     > Tool output with variable format  ← Fragile, avoid
```

## 8. Comprehension Check Results

### Question 1
**Asked**: "In your own words — what is the Stop hook actually doing in this loop? What problem would occur if it didn't exist?"

**Student Response**: Chose "You explain it" (model answer requested)

**Assessment**: Deferred — student requested model answer rather than independent attempt

**Model Answer Delivered**:
The Stop hook acts as an interception layer at the session boundary. Without it, Claude completes one pass and exits — the loop lives in the human's hands. The hook hijacks the exit moment, forces self-evaluation against the completion promise, and either allows exit (promise found) or reinjects a continuation prompt (promise not found). Key precision: the hook fires only at the exit boundary, not while Claude is working mid-iteration.
