The Problem

Waiting Time

You sit idle while Claude works. Minutes per cycle compound into hours across a session.

Context Switching

Technical Anatomy

The Four Technical Components

Stop Hook

Intercepts Claude's exit to reinject the loop prompt. It fires after every tool use, checking whether to continue or stop.

Completion Promise

A static done-signal. Exact string match. Set once, never changes. The ONLY way the loop terminates cleanly.

Max Iterations

Safety ceiling that prevents infinite loops. Always set this. It's your escape hatch when convergence fails.

Loop Prompt

The continuation template injected each cycle. Tells Claude what to check, how to verify, and what constraints to respect.

Key Pattern

Natural Tool Output

Grep for "0 errors" in build output. Fragile -- format changes, tool upgrades, and locale differences silently break the match.

Application

When to Use Ralph Loop

1. Framework Upgrades▼

Deterministic build output makes promises reliable. Upgrade Next.js, fix type errors iteratively -- the build either passes or it doesn't. Clear convergence signal.

2. Test-Driven Refactoring BEST▼

The gold standard. Tests ARE the specification. Claude can't game passing tests without actually fixing code. The promise is "all tests pass" -- objective, verifiable, and tamper-resistant.

3. Linting / Type Errors▼

Works well but has the highest gaming risk. Claude may add @ts-ignore or eslint-disable comments to "fix" errors without fixing root causes. Requires anti-gaming constraints in prompt.

4. Deployment Debugging▼

Riskiest valid use case. External state (servers, APIs) can change between iterations. Side effects aren't contained. Use with extreme caution and tight max-iteration limits.

Boundaries

When NOT to Use Ralph Loop

Human Judgment Tasks

Subjective quality -- "is this design good?" has no CLI verification.

Exploratory Research

Divergent by nature. No single done-state. Claude will loop forever or declare false completion.

Creative Work

No objective ceiling. "Write a better poem" has no CLI-verifiable endpoint.

Multi-Goal Tasks

Single-promise bottleneck. Ralph Loop has ONE completion signal -- multiple goals need decomposition.

External Input Tasks

No pause mechanism. Ralph Loop can't wait for user input mid-loop or poll external APIs.

Objective (use loop) Subjective (don't)

Risk Analysis

How Claude "Cheats"

@ts-ignore

Suppresses TypeScript errors instead of fixing them

any type

Replaces strict types with "any" to pass checks

eslint-disable

Disables lint rules instead of satisfying them

Deleting Code

Removes failing tests or problematic functions entirely

Changing Config

Loosens tsconfig or eslint config to allow violations

CONSTRAINTS (NEVER do these): - Do NOT add @ts-ignore or eslint-disable comments - Do NOT use 'any' type assertions - Do NOT delete existing tests - Do NOT modify tsconfig.json or .eslintrc - Do NOT remove code to make errors disappear

Always constrain what Claude must NOT do in the loop prompt.

Failure Analysis

What Goes Wrong

Cascading Fix Spiral▼

Coupled errors oscillate. Fixing A breaks B, fixing B breaks A. The loop bounces A→B→A→B forever. Root cause: errors share dependencies. Solution: decouple or fix at the shared root.

Scope Creep Accumulator▼

Claude "improves" beyond the task scope. It refactors code you didn't ask about, adds features, or reorganizes files. Each iteration drifts further from the original goal. Solution: constrain scope in loop prompt.

Silent Corruption▼

Technically passes all checks but is architecturally broken. Code works but is unmaintainable, has security holes, or violates patterns. The promise was too narrow to catch quality issues.

Terminology

Key Terms

Hover or tap cards to reveal definitions

Iteration Fatigue
Problem
Cognitive and time cost of manually shepherding Claude through repetitive fix-check cycles. Threshold: ~10 iterations.
Stop Hook
Component
A hook that intercepts Claude's exit event to reinject the loop prompt, enabling autonomous continuation without human intervention.
Completion Promise
Component
A static, exact-match done-signal that the loop checks each cycle. The sole termination criterion. Must be deterministic.
Max Iterations
Safety
A hard ceiling on loop cycles. Prevents infinite loops when convergence fails. Always set this as a safety mechanism.
Loop Prompt
Component
The continuation template reinjected each cycle. Contains task instructions, verification commands, and behavioral constraints.
Embedded Promise
Pattern
A pattern where Claude writes the completion signal to a file you control, rather than relying on external tool output. Recommended approach.
Ralph Loop
System
An autonomous iteration workflow where a stop hook, loop prompt, completion promise, and max iterations combine to let Claude work without human intervention.
